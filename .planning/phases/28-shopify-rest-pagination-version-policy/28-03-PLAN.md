Extend parseShopifyApiVersion with month-range validation and a sunset-versions reject-list, then update both REST and GraphQL route handlers to return distinguishable error responses.

Purpose: The current version regex /^(unstable|\d{4}-\d{2})$/ accepts syntactically-valid-but-nonsensical versions like 2024-99 and sunset versions like 2023-01 equally. Adding a SUNSET_VERSIONS Set catches the known-sunset versions. The error thrown for sunset versions carries an err.sunset=true flag so route handlers can return a different message (410 semantics surfaced as 400 with distinct error text).

Target implementation for api-version.ts:
```typescript
// Sunset = versions that were valid but are no longer supported
// As of 2026-03: Shopify's 12-month window means 2023 quarterly versions are sunset
const SUNSET_VERSIONS = new Set<string>([
  '2023-01', '2023-04', '2023-07', '2023-10',
]);

// Month validation: rejects 2024-99, 2024-00, etc.
const VALID_MONTH_RE = /^(0[1-9]|1[0-2])$/;

export function parseShopifyApiVersion(raw: string | undefined): string {
  if (!raw || !SHOPIFY_API_VERSION_RE.test(raw)) {
    throw new TypeError(`Invalid Shopify API version: "${raw}".`);
  }
  if (raw !== 'unstable') {
    const month = raw.split('-')[1];
    if (!VALID_MONTH_RE.test(month)) {
      throw new TypeError(`Invalid Shopify API version: "${raw}". Month out of range.`);
    }
  }
  if (SUNSET_VERSIONS.has(raw)) {
    const err = new Error(`Sunset Shopify API version: "${raw}".`) as any;
    err.sunset = true;
    throw err;
  }
  return raw;
}
```
