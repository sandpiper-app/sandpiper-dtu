    1. **EventDispatcher** (`src/services/event-dispatcher.ts`):
    - Constructor accepts `{ webhookQueue, slackStateManager, signingSecret }`.
    - `async dispatch(eventType: string, eventData: Record<string, unknown>): Promise<void>`:
      - Query slackStateManager.listEventSubscriptions() to find subscribers.
      - For each subscription whose event_types includes the eventType (or has '*'):
        - Build event envelope (from research Pattern 2):
          ```typescript
          {
            token: 'twin-verification-token',
            team_id: 'T_TWIN',
            api_app_id: subscription.app_id,
            event: { type: eventType, ...eventData, event_ts: String(Math.floor(Date.now() / 1000)) },
            type: 'event_callback',
            event_id: `Ev${generateSlackId('Ev').slice(2)}`,  // or just generateSlackId('Ev')
            event_time: Math.floor(Date.now() / 1000),
            authorizations: [{ enterprise_id: 'E0', team_id: 'T_TWIN', user_id: 'U_BOT_TWIN', is_bot: true }]
          }
          ```
        - Enqueue via webhookQueue.enqueue({ id: randomUUID(), topic: `slack:${eventType}`, callbackUrl: subscription.request_url, payload: envelope, secret: signingSecret, headers: { 'Content-Type': 'application/json' } }).
    - Use HMAC-SHA256 signing via @dtu/webhooks (it handles signing using the `secret` field). The signature format for Slack uses `v0:timestamp:body` basestring — BUT since our @dtu/webhooks uses standard HMAC signing, and this is a twin (not real Slack), use the same signing mechanism. The twin's signing is for test verification, not production Slack SDK compatibility. Use the webhook queue's built-in signing.

    2. **Events API endpoint** (`src/plugins/events-api.ts`):
    - `POST /events` — the endpoint apps configure as their Request URL.
    - Handle `url_verification` challenge:
      - If body.type === 'url_verification', immediately return `{ challenge: body.challenge }` with Content-Type application/json. Must respond in < 3 seconds (trivial for twin since it's local).
    - Handle `event_callback`:
      - Log received event for debugging.
      - Return 200 OK (this endpoint is for RECEIVING events from apps, not sending — the twin sends events via EventDispatcher TO apps).
      - Note: In real Slack, this endpoint is on the APP side. For the twin, this endpoint exists so apps can register/verify their Request URL against the twin. The twin as "Slack server" SENDS events via EventDispatcher; it also provides this endpoint for the url_verification handshake.

    3. **Wire EventDispatcher into chat.postMessage (with app_mention detection)**:
    - After successfully creating a message in chat.postMessage, call `eventDispatcher.dispatch('message', { channel: channelId, user: userId, text, ts, channel_type: 'channel' })`.
    - This triggers Events API delivery to all subscribed apps.
    - **Bot mention detection:** After dispatching the `message` event, check if the message text contains a bot mention pattern (regex `/<@U_BOT_TWIN>/` or more broadly `/<@U[A-Z0-9_]+>/` matching the twin's bot user ID). If a bot mention is found, additionally dispatch an `app_mention` event:
      ```typescript
      if (text && text.includes('<@U_BOT_TWIN>')) {
        eventDispatcher.dispatch('app_mention', {
          type: 'app_mention',
          user: userId,
          text,
          ts,
          channel: channelId,
          channel_type: 'channel',
        });
      }
      ```
    - This mirrors real Slack behavior where a `message` event AND an `app_mention` event are both dispatched when a message mentions the bot.
    - Do NOT await the dispatch (fire-and-forget, like real Slack). Unless WEBHOOK_SYNC_MODE is true (for tests).

    4. **Add admin endpoint for reactions** (needed for reaction_added events):
    - Add `POST /admin/reactions/add` to admin plugin (or create a small addition):
      - Body: `{ message_ts, channel, user_id, reaction }`
      - Store via slackStateManager.addReaction(message_ts, channel, user_id, reaction).
      - Dispatch `reaction_added` event: `eventDispatcher.dispatch('reaction_added', { type: 'reaction_added', user: user_id, reaction, item: { type: 'message', channel, ts: message_ts }, item_user: messageAuthorId, event_ts: String(Date.now()/1000) })`.
      - Return: `{ ok: true }`.

    5. **Register EventDispatcher in index.ts**:
    - Instantiate after webhookQueue and slackStateManager are created.
    - Decorate on fastify as `eventDispatcher`.
    - Update FastifyInstance type augmentation.
    - Register eventsApiPlugin.

    1. **InteractionHandler** (`src/services/interaction-handler.ts`):
    - Constructor accepts `{ slackStateManager, signingSecret }`.
    - Maintains a `Map<string, ResponseUrlEntry>` for response URLs, where ResponseUrlEntry = `{ channelId, messageTs, usesRemaining: 5, expiresAt: Date.now() + 30*60*1000 }`.
    - `generateInteractionPayload(opts: { messageTs, channelId, actionId, userId, blockId? }): { payload: BlockActionsPayload, responseUrl: string }`:
      - Look up message via slackStateManager.getMessage(messageTs).
      - Look up channel via slackStateManager.getChannel(channelId).
      - Generate response URL: `/response-url/{randomId}` (twin-local path).
      - Store response URL entry in map.
      - Build block_actions payload (from research):
        ```typescript
        {
          type: 'block_actions',
          trigger_id: `${Date.now()}.${Math.floor(Math.random()*1e9)}`,
          user: { id: userId, username: userName, name: realName },
          team: { id: 'T_TWIN', domain: 'twin-workspace' },
          api_app_id: 'A_TWIN',
          token: 'twin-verification-token',
          container: { type: 'message', message_ts: messageTs, channel_id: channelId },
          channel: { id: channelId, name: channelName },
          message: { /* original message object */ },
          actions: [{ type: 'button', block_id: blockId ?? 'actions_block', action_id: actionId, text: { type: 'plain_text', text: actionId }, value: actionId, action_ts: String(Date.now()/1000) }],
          response_url: responseUrl
        }
        ```
      - Return { payload, responseUrl }.
    - `handleResponseUrl(responseUrlId: string, body: any): { ok: boolean; error?: string }`:
      - Look up entry in map. If not found or expired, return `{ ok: false, error: 'expired_url' }`.
      - If usesRemaining <= 0, return `{ ok: false, error: 'expired_url' }`.
      - Decrement usesRemaining.
      - Post the body content as a message in the original channel (via slackStateManager.createMessage).
      - Return `{ ok: true }`.

    2. **Interactions plugin** (`src/plugins/interactions.ts`):
    - `POST /admin/interactions/trigger` — admin endpoint to simulate a user clicking a button:
      - Body: `{ message_ts, channel, action_id, user_id, block_id? }`
      - Call interactionHandler.generateInteractionPayload(opts).
      - Look up event subscriptions (or a dedicated interaction URL config — use the first event subscription's request_url or a configurable INTERACTION_URL env var).
      - Deliver payload to app's Request URL as `application/x-www-form-urlencoded` with body `payload={urlEncodedJson}` (CRITICAL Pitfall 8 from research: interaction payloads use form-encoded, not JSON).
      - Use native fetch() or webhookQueue to deliver. For simplicity, deliver synchronously with fetch() since this is admin-triggered.
      - Return: `{ ok: true, response_url: responseUrl }`.

    - `POST /response-url/:id` — response URL endpoint:
      - Extract id from params.
      - Parse JSON body.
      - Call interactionHandler.handleResponseUrl(id, body).
      - Return result.
