    Create `twins/slack/` package mirroring twins/shopify structure:

    1. **package.json**: Name `@dtu/twin-slack`, type module, dependencies on `@dtu/core`, `@dtu/state`, `@dtu/types`, `@dtu/webhooks`, `fastify@^5.0.0`, `pino@^9.5.0`, `pino-pretty@^13.0.0`. Dev deps: `@types/node`, `tsx`, `typescript`, `vitest`. Scripts: build, dev, start, test (same pattern as Shopify twin but no graphql deps).

    2. **tsconfig.json**: Extend `../../tsconfig.base.json`, composite true, outDir dist, rootDir src, references to `../../packages/types`, `../../packages/state`, `../../packages/core`, `../../packages/webhooks`.

    4. **SlackStateManager** (`src/state/slack-state-manager.ts`): Composition wrapper around StateManager (not inheritance). Constructor accepts `{ dbPath?: string }`. Internally creates and initializes StateManager. Exposes `database` getter, `reset()`, `close()` delegating to inner StateManager.

    Run Slack-specific migrations creating these tables (from research):
    - `slack_teams` (id TEXT PK, name TEXT, domain TEXT)
    - `slack_channels` (id TEXT PK, name TEXT, is_channel BOOLEAN, is_private BOOLEAN, is_archived BOOLEAN, topic TEXT, purpose TEXT, creator TEXT, num_members INTEGER, created INTEGER)
    - `slack_users` (id TEXT PK, team_id TEXT, name TEXT, real_name TEXT, display_name TEXT, email TEXT, is_admin BOOLEAN, is_bot BOOLEAN, deleted BOOLEAN, color TEXT, tz TEXT)
    - `slack_messages` (id INTEGER PK AUTOINCREMENT, channel_id TEXT, user_id TEXT, text TEXT, blocks TEXT, ts TEXT UNIQUE, thread_ts TEXT, subtype TEXT, edited_user TEXT, edited_ts TEXT, created_at INTEGER)
    - `slack_tokens` (token TEXT PK, token_type TEXT, team_id TEXT, user_id TEXT, scope TEXT, app_id TEXT, created_at INTEGER)
    - `slack_event_subscriptions` (id INTEGER PK AUTOINCREMENT, app_id TEXT, request_url TEXT, event_types TEXT, created_at INTEGER)
    - `slack_error_configs` (id INTEGER PK AUTOINCREMENT, method_name TEXT UNIQUE, status_code INTEGER, error_body TEXT, delay_ms INTEGER, enabled BOOLEAN DEFAULT 1)
    - `slack_reactions` (id INTEGER PK AUTOINCREMENT, message_ts TEXT, channel_id TEXT, user_id TEXT, reaction TEXT, created_at INTEGER)

    Prepared statement methods:
    - `createTeam(id, name, domain)`, `getTeam(id)`
    - `createChannel(data)`, `getChannel(id)`, `listChannels()`, `updateChannel(id, data)`
    - `createUser(data)`, `getUser(id)`, `listUsers()`
    - `createMessage(data)` returning message row, `getMessage(ts)`, `listMessages(channelId, opts?)` with limit/cursor support, `updateMessage(ts, data)`
    - `createToken(token, tokenType, teamId, userId, scope, appId)`, `getToken(token)`, `listTokens()`
    - `createEventSubscription(appId, requestUrl, eventTypes)`, `listEventSubscriptions()`
    - `createErrorConfig(methodName, config)`, `getErrorConfig(methodName)`, `clearErrorConfigs()`
    - `addReaction(messageTs, channelId, userId, reaction)`, `listReactions(messageTs)`

    Use prepared statements for all queries (same performance pattern as Shopify's StateManager). All prepared statements must be nullified in reset() and close().

    5. **ID Generator** (`src/services/id-generator.ts`):
    - `generateSlackId(prefix: string): string` — generates `{prefix}{9 alphanumeric chars}` using uppercase alphanumeric charset (0-9A-Z)
    - `generateMessageTs(): string` — returns `{epoch}.{6-digit-counter}` format. Use a module-level counter that increments per call.
    - Export convenience functions: `generateTeamId()`, `generateChannelId()`, `generateUserId()`, `generateAppId()` calling generateSlackId with T, C, U, A prefixes respectively.

    1. **buildApp()** (`src/index.ts`): Follow Shopify twin's index.ts pattern exactly:
    - Create Fastify instance with Pino logger, x-request-id header, UUID request IDs
    - Instantiate SlackStateManager with DB_PATH env var (default ':memory:')
    - Instantiate SqliteDeadLetterStore sharing slackStateManager.database
    - Instantiate WebhookQueue with WEBHOOK_TIME_SCALE, WEBHOOK_SYNC_MODE env vars
    - Configure signingSecret from SLACK_SIGNING_SECRET env var (default 'dev-signing-secret')
    - Decorate fastify with: slackStateManager, webhookQueue, deadLetterStore, signingSecret
    - Register health, admin, oauth plugins
    - Add onClose hook: webhookQueue.shutdown(), deadLetterStore.close(), slackStateManager.close()
    - Export buildApp and add isMainModule start logic (same as Shopify twin, default port 3001)

    Declare module augmentation for FastifyInstance:
    ```typescript
    declare module 'fastify' {
      interface FastifyInstance {
        slackStateManager: SlackStateManager;
        webhookQueue: WebhookQueue;
        deadLetterStore: DeadLetterStore;
        signingSecret: string;
      }
    }
    ```

    2. **Health plugin** (`src/plugins/health.ts`): GET /health returning `{ status: 'ok', twin: 'slack', uptime: process.uptime() }`. Same pattern as Shopify.

    3. **Admin plugin** (`src/plugins/admin.ts`):
    - `POST /admin/reset` — calls slackStateManager.reset(), returns `{ reset: true, timestamp: Date.now() }`. Note: does NOT clear DLQ (same issue as Shopify — reset closes DB, invalidating DLQ store reference).
    - `POST /admin/fixtures/load` — accepts `{ channels?, users?, messages?, teams? }` arrays. For each entity, generate appropriate Slack ID (using id-generator) if not provided, call slackStateManager.create* method. Return loaded counts.
    - `GET /admin/state` — return counts for channels, users, messages, tokens, event_subscriptions.
    - DLQ endpoints: GET/DELETE /admin/dead-letter-queue (list/clear), GET/POST/DELETE /admin/dead-letter-queue/:id (get/retry/remove) — same pattern as Shopify admin.ts.

    4. **OAuth plugin** (`src/plugins/oauth.ts`):
    - `GET /oauth/v2/authorize` — simplified redirect flow. Accepts query params: client_id, scope, redirect_uri, state. Redirects to redirect_uri with `code={generated-code}&state={state}`.
    - `POST /api/oauth.v2.access` — token exchange. Accepts body/form: code, client_id, client_secret. Generates bot token (`xoxb-{teamId}-{random}`) and user token (`xoxp-{teamId}-{random}`). Stores both in slackStateManager.createToken(). Returns Slack-format response:
    ```json
    {
      "ok": true,
      "access_token": "xoxb-...",
      "token_type": "bot",
      "scope": "chat:write,channels:read,...",
      "bot_user_id": "U_BOT_TWIN",
      "app_id": "A_TWIN",
      "team": { "name": "Twin Workspace", "id": "T_TWIN" },
      "authed_user": { "id": "U_AUTHED", "scope": "...", "access_token": "xoxp-...", "token_type": "user" }
    }
    ```
    Ensure default team (T_TWIN, "Twin Workspace", "twin-workspace") is seeded in slack_teams on first oauth exchange if not present.

    Also create a default team and bot user during SlackStateManager.init() so the twin is usable immediately:
    - Team: id='T_TWIN', name='Twin Workspace', domain='twin-workspace'
    - Bot user: id='U_BOT_TWIN', team_id='T_TWIN', name='twin-bot', real_name='Twin Bot', is_bot=true
    - General channel: id='C_GENERAL', name='general', creator='U_BOT_TWIN'

    This seeding should happen in SlackStateManager.init() after table creation (and after reset, since reset re-inits).
