    ## Phase B: Fix Known Discrepancies

    ### B1. Unified token extraction (token-validator.ts)

    The real Slack API accepts auth tokens via three mechanisms:
    1. `Authorization: Bearer xoxb-...` header (preferred)
    2. `token=xoxb-...` in POST body parameter
    3. `token=xoxb-...` in query string parameter

    Our twin only checks (1). Update `extractBearerToken` (or create a new `extractToken` function) to check all three in priority order:

    ```typescript
    /**
     * Extract API token from request. Checks (in order):
     * 1. Authorization: Bearer header
     * 2. token field in request body
     * 3. token query parameter
     *
     * Matches real Slack Web API auth behavior.
     */
    export function extractToken(request: FastifyRequest): string | null {
      // 1. Bearer header (preferred)
      const auth = request.headers.authorization;
      if (auth?.startsWith('Bearer ')) {
        return auth.slice(7);
      }

      // 2. Body token param (form-urlencoded or JSON)
      const body = request.body as Record<string, any> | null;
      if (body?.token && typeof body.token === 'string') {
        return body.token;
      }

      // 3. Query string token param
      const query = request.query as Record<string, any>;
      if (query?.token && typeof query.token === 'string') {
        return query.token;
      }

      return null;
    }
    ```

    Keep the old `extractBearerToken` as a deprecated alias if other code references it, or update all call sites to use `extractToken`. Update all Web API route handlers to use the new function.

    ### B2. GET support for Slack read methods (conversations.ts, users.ts)

     * Extract API parameters from GET query string or POST body.

    function getParams(request: FastifyRequest): Record<string, any> {
      if (request.method === 'GET') {
        return (request.query as Record<string, any>) ?? {};

    }

    Also update chat.ts handlers to use `extractToken(request)` instead of `extractBearerToken(request)`.

    ### B4. Slack oauth.v2.access form-urlencoded support (oauth.ts)

    **Critical:** The real Slack `oauth.v2.access` endpoint specifically REQUIRES `application/x-www-form-urlencoded` — it does NOT accept JSON. Our twin does the opposite (accepts JSON, not form-urlencoded).

    With `@fastify/formbody` registered, Fastify will parse form-urlencoded POST bodies into `request.body`. The current handler reads from `request.body` so it should work for both formats without code changes — verify this is the case. If the handler has type-specific logic, update it to handle both.
