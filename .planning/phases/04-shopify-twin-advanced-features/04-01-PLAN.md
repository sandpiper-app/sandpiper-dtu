Create `twins/shopify/src/services/query-cost.ts`:
- Export `calculateQueryCost(document: DocumentNode, schema: GraphQLSchema, variables?: Record<string, unknown>): number`
- Use `graphql`'s `visit()` function to traverse the AST
- Cost rules (Shopify algorithm):
  - Scalar/Enum fields: 0 points
  - Object fields: 1 point (multiply by parent connection size if nested)
  - Connection fields (detected by having `first` or `last` argument in the schema, or field name ending in `Connection` type): 2 + (first ?? last ?? 10) points
  - Mutations: 10 points base cost
- Track a multiplier stack for nested connections: when entering a connection field, push its page size (first/last arg value) onto stack; child costs are multiplied by this. When leaving, pop.
- Use schema introspection (`schema.getType()`, `schema.getQueryType()`) to determine if a field returns an object type vs scalar. For the twin's fixed schema, detect connections by checking if the field type name ends with "Connection" or has `edges` field.
- Handle variables: if `first` argument references a variable (`$first`), resolve from `variables` parameter.

Create `twins/shopify/src/services/rate-limiter.ts`:
- Export `LeakyBucketRateLimiter` class
- Constructor takes `maxAvailable` (default 1000) and `restoreRate` (default 50 pts/sec)
- `tryConsume(key: string, cost: number)` returns `{ allowed: boolean; currentlyAvailable: number; retryAfterMs: number }`
- Refill logic: on each call, calculate elapsed time since last refill, add `elapsed * restoreRate` to available (capped at maxAvailable)
- If `available >= cost`: deduct and return allowed=true
- If `available < cost`: calculate deficit, return allowed=false with retryAfterMs = ceil((deficit / restoreRate) * 1000)
- `reset()` method clears all buckets (for /admin/reset)
- Expose `maxAvailable` and `restoreRate` as readonly public properties (needed for throttled response body)
- Internal state: `Map<string, { available: number; lastRefill: number }>`

Create unit tests for both services:
- query-cost.test.ts: test simple scalar query (cost ~0-2), single connection (cost = 2+first), nested connections (cost multiplied), mutation (10 base), query with variables
- rate-limiter.test.ts: test basic consumption, bucket refill over time (use small time advances), throttling when depleted, reset clears state

- Create a module-level `LeakyBucketRateLimiter` instance inside the plugin (shared across requests). Alternatively, attach to Fastify instance as a decorator so admin.ts can access it for reset.
- In the route handler, BEFORE `yoga.fetch()`:
  1. Parse the request body to extract `query` and `variables`
  2. If `query` exists, `parse(query)` to get DocumentNode, then `calculateQueryCost(document, schema, variables)` to get cost
  3. Extract rate limit key from `x-shopify-access-token` header (fallback to 'anonymous')
  4. Call `rateLimiter.tryConsume(key, cost)`
  5. If NOT allowed: return HTTP 429 with `Retry-After` header (seconds, ceil of retryAfterMs/1000) and Shopify-format JSON body:
     ```json
     { "errors": [{"message": "Throttled"}], "extensions": { "cost": { "requestedQueryCost": <cost>, "actualQueryCost": null, "throttleStatus": { "maximumAvailable": 1000.0, "currentlyAvailable": <available>, "restoreRate": 50.0 }}}}
     ```
  6. If allowed: proceed with yoga.fetch(), then parse the response text, inject `extensions.cost` into the response JSON before sending. The cost extension for successful requests:
     ```json
     { "requestedQueryCost": <cost>, "actualQueryCost": <cost>, "throttleStatus": { "maximumAvailable": 1000.0, "currentlyAvailable": <remaining>, "restoreRate": 50.0 }}
     ```
- Register the rateLimiter as a Fastify decorator: `fastify.decorate('rateLimiter', rateLimiter)` so admin plugin can call reset.
- Add to the FastifyInstance type declaration (in admin.ts or a shared type file): `rateLimiter: LeakyBucketRateLimiter`
