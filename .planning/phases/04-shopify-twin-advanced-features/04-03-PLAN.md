Create `twins/shopify/src/services/order-lifecycle.ts`:
- Define types:
  ```typescript
  type FulfillmentStatus = 'UNFULFILLED' | 'IN_PROGRESS' | 'PARTIALLY_FULFILLED' | 'FULFILLED';
  type FinancialStatus = 'PENDING' | 'AUTHORIZED' | 'PAID' | 'PARTIALLY_REFUNDED' | 'REFUNDED';
  ```
- Export `validateFulfillment(order: { fulfillmentStatus: FulfillmentStatus; closedAt: number | null }): string | null`
  - If `fulfillmentStatus === 'FULFILLED'` return `'Order is already fulfilled'`
  - If `closedAt !== null` return `'Cannot fulfill a closed order'`
  - Otherwise return null (allowed)
- Export `validateClose(order: { fulfillmentStatus: FulfillmentStatus; financialStatus: FinancialStatus; closedAt: number | null }): string | null`
  - If `closedAt !== null` return `'Order is already closed'`
  - If `fulfillmentStatus !== 'FULFILLED'` return `'Order must be fully fulfilled before closing'`
  - If financialStatus not in `['PAID', 'PARTIALLY_REFUNDED', 'REFUNDED']` return `'Order must have completed financial transactions before closing'`
  - Otherwise return null (allowed)
- Export the type aliases for use in resolvers

Update `twins/shopify/src/schema/resolvers.ts`:
- Import `validateFulfillment`, `validateClose` from `../services/order-lifecycle.js`
- Add `Order.displayFulfillmentStatus` type resolver: `(parent) => parent.display_fulfillment_status || 'UNFULFILLED'`
- Add `Order.displayFinancialStatus` type resolver: `(parent) => parent.display_financial_status || 'PENDING'`
- Add `Order.closedAt` type resolver: `(parent) => parent.closed_at`

- Modify `fulfillmentCreate` mutation resolver:
  1. After creating fulfillment, look up the parent order using order_gid
  2. Call `validateFulfillment({ fulfillmentStatus: order.display_fulfillment_status, closedAt: order.closed_at })`
  3. If validation fails, return userErrors (but still create the fulfillment -- Shopify allows partial fulfillments to exist even if status update is rejected, though for the twin's simplified model, reject the entire operation with userErrors and DON'T create the fulfillment)
  4. If validation passes: call `context.stateManager.updateOrderFulfillmentStatus(orderId, 'FULFILLED')`
  5. After status update, also enqueue `orders/update` webhook with updated order data (status change triggers webhook)
  - Parse order GID to get numeric id for the stateManager call

- Add `orderClose` mutation resolver:
  1. requireAuth(context), errorSimulator check
  2. Parse input.id GID to get numeric order ID
  3. Look up order, return userErrors if not found
  4. Call `validateClose({ fulfillmentStatus: order.display_fulfillment_status, financialStatus: order.display_financial_status, closedAt: order.closed_at })`
  5. If validation fails, return `{ order: null, userErrors: [{ field: ['id'], message: error }] }`
  6. If passes: call `context.stateManager.closeOrder(orderId)`
  7. Fetch updated order, enqueue `orders/update` webhook
  8. Return `{ order: closedOrder, userErrors: [] }`

- Modify `orderCreate` resolver to pass `financialStatus` (or default 'PENDING') to stateManager.createOrder
