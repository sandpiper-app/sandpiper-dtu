Fix the two flaky DLQ timing tests that use `setTimeout(600)` by replacing them with polling loops that exit immediately when the DLQ entry appears.

**Fix 1: Line 970 (failed webhook deliveries test)**

In the test "failed webhook deliveries appear in dead letter queue after retries (compressed timing)" (~line 947), replace line 970:

```typescript
      // Wait for all retries to complete (360ms at 0.001x scale + buffer)
      await new Promise(resolve => setTimeout(resolve, 600));
```

With:

```typescript
      // Poll for DLQ entry instead of fixed wait — eliminates race condition under CI load
      const deadline = Date.now() + 5000;
      while (Date.now() < deadline) {
        const pollRes = await app.inject({ method: 'GET', url: '/admin/dead-letter-queue' });
        if (JSON.parse(pollRes.body).length > 0) break;
        await new Promise(resolve => setTimeout(resolve, 50));
      }
```

**Fix 2: Line 1027 (DELETE /admin/dead-letter-queue test)**

In the test "DELETE /admin/dead-letter-queue clears the queue" (~line 995), replace line 1027:

```typescript
      // Wait for retries to complete
      await new Promise(resolve => setTimeout(resolve, 600));
```

With:

```typescript
      // Poll for DLQ entry instead of fixed wait — eliminates race condition under CI load
      const deadline = Date.now() + 5000;
      while (Date.now() < deadline) {
        const pollRes = await app.inject({ method: 'GET', url: '/admin/dead-letter-queue' });
        if (JSON.parse(pollRes.body).length > 0) break;
        await new Promise(resolve => setTimeout(resolve, 50));
      }
```

Both fixes use the same pattern:
1. Poll DLQ at 50ms intervals (fast enough for test speed)
2. 5-second hard timeout (generous for any CI load)
3. Exit immediately when DLQ entry appears (no unnecessary waiting)
4. Existing assertions after the loop remain unchanged -- if the deadline passes without a DLQ entry, the assertions fail with a clear message

Do NOT use `vitest.waitFor` -- the DLQ check crosses process boundaries via HTTP inject, and polling with short intervals is cleaner.

Do NOT change the webhook time scale or retry configuration.
