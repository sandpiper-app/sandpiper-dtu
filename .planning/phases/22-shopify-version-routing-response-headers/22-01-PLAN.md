version-aware.

response, and preserve the requested version in pagination `Link` headers.

<task type="auto">
  <name>Task 1: Add shared Shopify API version helper and refactor GraphQL routes to :version wrappers</name>
  <files>twins/shopify/src/services/api-version.ts, twins/shopify/src/plugins/graphql.ts</files>
  <action>
Create `twins/shopify/src/services/api-version.ts` with the shared transport utilities that both
GraphQL and REST can call. Keep it small and explicit. It should include:

- A parser/validator for route params:
  - Accept `unstable`
  - Accept `YYYY-MM` strings such as `2024-01`, `2025-01`, `2025-07`
  - Reject obviously invalid values with a normal API-style error response path
- A helper for setting `X-Shopify-API-Version` on replies
- A helper for building admin paths such as `/admin/api/${version}/products.json`

Then refactor `twins/shopify/src/plugins/graphql.ts`:

1. Keep the existing Yoga instance and canonical endpoint at `/admin/api/2024-01/graphql.json`.
   Do NOT create one Yoga instance per version.
2. Replace the fixed admin route with `/admin/api/:version/graphql.json`.
3. Replace the fixed Storefront route with `/api/:version/graphql.json`.
4. At the top of each handler:
   - Parse `req.params.version`
   - Set `X-Shopify-API-Version` on the reply before auth/throttle branches
5. Preserve existing auth and rate-limiter behavior.
6. When calling `yoga.fetch()`, rewrite the request URL to the canonical admin route so Yoga still
   executes correctly.
7. Ensure 401, 429, GraphQL JSON responses, and non-JSON pass-through responses all still include
   the echoed `X-Shopify-API-Version` header.

Do NOT change GraphQL schema/resolvers in this plan unless a compile/runtime need forces it. This is
a transport-layer refactor, not a schema-versioning phase.
  </action>

    - Successful and error API responses both emit `X-Shopify-API-Version`
